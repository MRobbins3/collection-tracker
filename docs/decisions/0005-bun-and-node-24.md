# ADR 0005: Web toolchain — Bun as package manager, Node 24 as runtime target

- **Status:** Accepted (directional — migration scheduled as Milestone 11, before CI)
- **Date:** 2026-04-19

## Context

The web app was scaffolded with pnpm + Node 20 because that's what we had in hand on day one. We now have a running stack and a clear roadmap; it's cheaper to migrate the toolchain before CI hardens around the current choices than after.

Two separate questions came up together:

1. Can we switch the web package manager from pnpm to Bun?
2. Should we pin the Node runtime with an `.nvmrc`, and move to Node 24?

## Decision

Yes to both. Land them together as Milestone 11, ahead of Milestone 12 (GitHub Actions CI) so CI is built once against the final toolchain.

### Scope (what the migration milestone changes)

**Package manager (pnpm → Bun):**
- Replace `web/pnpm-lock.yaml` with `web/bun.lockb` (regenerated by `bun install`).
- Update `web/package.json`: `packageManager` → `bun@1.x`.
- Update `docker-compose.yml` `web` service:
  - image `node:20-alpine` → `oven/bun:1`
  - command `corepack enable && pnpm install && pnpm dev --host 0.0.0.0` → `bun install && bun run dev --host 0.0.0.0`
- Update scripts/docs to say `bun install`, `bun run test`, etc.
- Keep Vitest (not `bun:test`). Vue SFC + `@vitejs/plugin-vue` + `@vue/test-utils` + MSW all live in the Vitest ecosystem and that's not a migration we want to bundle in here.
- Keep Playwright. `bunx playwright install` works; the browsers install story is unchanged.

**Runtime pin (→ Node 24):**
- Add `.nvmrc` at repo root with `24`.
- After the Bun switch, the `web` container no longer runs Node at all, but developers who work outside Docker still need a pinned Node version for tooling, lint runners, and the occasional `node` script. `.nvmrc` documents the target.
- If we end up keeping a Node-based service later (unlikely but possible), bump its image to `node:24-alpine` in the same migration.
- Update `README.md` "Prerequisites" to say "Node 24+ (for non-Docker tooling)".

**Validation gate (what must still pass after the switch):**
- `bun install` clean.
- `bun run test` — all Vitest suites green.
- `bun run typecheck` — no new type errors.
- `docker compose up -d` — web serves on :3000, SSR works, hydration still matches.
- Hit the full set of pages: `/`, `/categories`, `/categories/lego-sets`, `/my`, `/my/:id`. Add an item with category-specific attributes. Rename and delete a collection.
- `bunx playwright install chromium` + running any future Playwright smoke works.

### Non-goals

- **Bun as the *runtime* for Nuxt** (not just the package manager). Nuxt's Bun-runtime support exists but is less battle-tested than Node; we'd be the early adopter. Revisit once we have a reason (perf, deploy simplification).
- **`bun:test` replacing Vitest.** The Vue SFC + plugin ecosystem assumes Vitest. Not worth the migration risk for MVP.
- **Replacing Go tooling.** This is web-only.

## Consequences

**Positive**
- Faster `install` (typically 3–10× over pnpm for cold installs).
- `bunx` replaces `pnpm exec` — fewer layers.
- `.nvmrc` stops the "which Node are you on?" drift when we onboard a second contributor.
- Doing this before CI means the GH Actions workflows only ever see the final toolchain.

**Negative / trade-offs**
- ~10% of Node-native packages still have bun rough edges. Our dependency tree (Nuxt + Tailwind + Vitest + Playwright) is mainstream, but we should still run the full test matrix after the switch as a gate.
- Bun's `bun.lockb` is a binary file; less greppable than `pnpm-lock.yaml` but diff tooling (`bun.lockb` has an explicit text view via `bun install --print` + text lockfile option) exists if it ever matters.
- Two tooling ecosystems in the repo (Go + Bun/Node). Already true today; no change.

## Revision triggers

- Bun disappears / stops moving (unlikely in the near term).
- A real Bun incompatibility surfaces that we can't work around.
- We decide to make Bun the Nuxt *runtime* too, which is a separate future decision.
